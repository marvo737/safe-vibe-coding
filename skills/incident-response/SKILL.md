# Incident Response

## Overview

本番事故発生時の最初の30分で被害拡大を止めるための定型手順。**パニックで誤った操作を追加しない** ことが最優先。

## When to Use

- データ漏洩・個人情報流出の疑い
- データ破損・誤削除
- 課金の急増(数時間で通常の10倍以上)
- サービス全体または主要機能の停止
- セキュリティ侵害(不正アクセス・マルウェア)
- 誤ったバッチ・マイグレーションの実行

**平常運用での軽微なバグ修正には使わない。**

## The 5-Step Sequence

### 1. 止める(Stop)— 最優先
目的: 被害の拡大を止める。原因究明より先。

- 影響機能をフィーチャーフラグでOFF
- 該当エンドポイントにメンテナンス応答
- 流出の疑いならAPIキー/シークレットを **即座にローテーション**
- 課金暴走ならサービスを一時停止 / IAM権限を剥奪
- **復旧手段を失う操作(DB DROP、削除系)は絶対にしない**

### 2. 記録する(Record)
目的: 後から時系列を再構築できるようにする。

- 共有チャンネル(Slack等)に専用スレを立て全行動を書き込む
- **時刻・誰が・何をしたか・なぜしたか** を全て残す
- スクリーンショット/ログを保存(後で消える可能性)
- 推測と事実を分けて書く(「〜らしい」と「〜を確認」)

### 3. 知らせる(Notify)
目的: 影響ユーザーと関係者への早期周知。

- 社内: 上長・セキュリティ担当・法務(個人情報関連時)
- ユーザー: 影響範囲・現在の状況・次の更新予定時刻
- **「調査中」で良いので早く出す**、沈黙が最悪
- 法定通知義務: 日本の個人情報保護法では、個情委への報告は **速報=事態を知った後速やかに(ガイドライン上の目安は概ね3〜5日)**、**確報=30日以内(不正アクセス等の意図的な漏えいは60日以内)**。本人通知も別途必要。海外居住者を含む場合は GDPR(72時間以内)等の適用も確認

### 4. 原因究明(Investigate)
目的: 再発を防ぐため **根本原因** を特定する。表面の症状で止めない。

- `Why?` を5回繰り返して遡る
- 直近の変更(デプロイ・設定変更・マイグレーション)を時系列で洗う
- ログ・メトリクス・トレースから事実を積む
- **「誰が悪い」ではなく「なぜそうなる仕組みだったか」**

### 5. 再発防止(Prevent)
目的: 人間の注意ではなく **仕組み** で防ぐ。

- テストケース追加(バグを再現するテスト)
- アラート追加(早期検知)
- 権限・手順の見直し(削除はレビュー必須 等)
- ポストモーテムを書いて共有(非難しない文化で)

## Red Flags — 事故中にやってはいけない

| やりがちな失敗 | 正しい行動 |
|---|---|
| 原因を探りながら本番で色々試す | まず止める。試行はステージング/コピーで |
| ログを消して「綺麗に」する | 証拠保全。触らない |
| 一人で抱え込む | 即エスカレ。複数人で記録 |
| ユーザー通知を「確定してから」 | 不確定でも速報を出す |
| 「誰のミスか」で時間を使う | 仕組みの問題として扱う |
| `DROP` / `rm -rf` / `git push --force` | 復旧手段を失う、絶対禁止 |

## Common Mistakes

- **Step 1 を飛ばして Step 4 に行く:** 調査中に被害が広がる。まず止める。
- **記録を後で書こうとする:** 記憶は歪む。リアルタイムに残す。
- **ポストモーテムで個人を非難:** 次の事故の隠蔽を生む。blameless に。

## Escalation Template

```
【事故速報】 YYYY-MM-DD HH:MM 発生
影響: (対象ユーザー / 対象機能)
現状: (止めた / 調査中 / 復旧中)
既知の事実: ...
不明点: ...
次回更新: HH:MM
担当: @name
```
